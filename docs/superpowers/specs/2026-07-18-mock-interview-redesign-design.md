# 模拟面试界面重构设计

状态：已由用户确认

日期：2026-07-18

## 1. 功能摘要

本次重构为面向大学生的求职工作台补全模拟面试流程。用户从面试首页查看已完成记录或创建新面试，在独立配置页选择岗位、类型、难度、预计时长和知识资料，然后进入持续语音监听的正式面试。AI 决定何时结束面试，用户完成最后一轮回答后进入总结页查看基于对话证据的评分与改进建议。

设计服务于任务本身。界面延续现有工作台的主题、导航、设计令牌和组件语言，保持可信、平静、鼓励，不使用数字人舞台、视频会议布局、游戏化积分或装饰性动效。

## 2. 用户与成功标准

主要用户是准备求职的大学生。桌面 Web 和 Electron 是第一优先，移动 Web 保证可以完成基本流程。

成功标准：

- 用户几乎不需要学习操作即可开始并完成一次模拟面试。
- 正式面试页始终清楚表达当前处于提问、监听、提交、思考、回复或结束中的哪一阶段。
- 语音转写保留真实表达过程，不允许编辑、撤回或用文字输入替代。
- 学生在总结中能理解每项评分来自哪些对话证据，并获得下一次可执行的练习建议。
- 页面不承诺尚未冻结的后端媒体、流式或评分协议。

## 3. 范围与非目标

### 3.1 本次范围

- 面试首页、配置页、正式面试页和总结页四个独立路由。
- 页面所需的领域模型、Gateway 能力和 Mock adapter 行为。
- 加载、空、失败、重试、重复提交、取消和卸载清理等前端状态。
- 中英文资源、深浅主题、桌面布局和基础移动布局。
- 基于确定性 Mock 的完整流程与自动化测试。

### 3.2 非目标

- 不接入真实麦克风采集、语音识别、WebRTC、WebSocket 或未冻结的面试流协议。
- 不编造 HTTP method、path、DTO、鉴权头、响应包装或报告 Job 协议。
- 不新增文字输入模式，不允许编辑或撤回实时转写。
- 不实现历史搜索、筛选、删除、重命名、分页或中断会话恢复。
- 不展示跨场次趋势、雷达图、人格、情绪或受保护属性推断。
- 不进行项目范围的无障碍改造；保留现有基础能力且不造成回退。

## 4. 信息架构与路由

采用四个独立页面：

```text
/interviews
  ├─ /interviews/new
  │    └─ /interviews/:sessionId
  │          └─ /interviews/:sessionId/summary
  └───────────────────────────────┘
             历史记录直接打开总结
```

- `/interviews`：新面试入口与已完成历史记录。
- `/interviews/new`：面试配置。
- `/interviews/:sessionId`：正式面试。
- `/interviews/:sessionId/summary`：总结生成状态与最终报告。

主导航的“面试”入口改为 `/interviews`，不再直接指向固定 Mock 会话。每个关键页面拥有可恢复的 URL 和可预测的浏览器返回行为。

## 5. 面试首页

### 5.1 布局

页面顶部使用紧凑的横向启动区，而不是大型宣传卡片：左侧为标题和一句说明，右侧为唯一主操作“开始新面试”。移动端上下排列，按钮占满可用宽度。

下方使用逐行历史列表，列为：

- 目标岗位
- 面试类型
- 完成时间
- 实际时长
- 总评分

整行可点击并进入对应总结，默认按完成时间倒序排列。第一版不提供行内更多菜单。

### 5.2 数据规则与状态

- 只展示状态为已完成且报告已成功生成的会话。
- 首次使用时显示说明性空状态和“开始第一次模拟面试”。
- 加载时使用与列表结构一致的骨架。
- 请求失败时显示原因可理解的错误与重试操作，不把失败误判为空记录。
- Mock 数据量保持在无需分页的范围；真实分页协议冻结前不伪造分页行为。

## 6. 新面试配置页

### 6.1 表单结构

使用单列、适中宽度表单，不采用多步骤向导。字段顺序为：

1. 目标岗位：必填，可选择已有求职目标或手动输入。
2. 面试类型：综合、行为、技术、系统设计。
3. 难度：基础、标准、进阶。
4. 预计时长：15、30、45、60 分钟。
5. 补充要求：可选的一句话。
6. 知识库资料：逐行展示当前资料，进入页面时默认全选，支持单独取消和一键全选。

面试类型第一版不根据岗位动态隐藏，避免产生不可解释的规则。公司名称、完整职位描述、题目数量和评分规则不单独暴露。

### 6.2 行为与状态

- 无知识资料时仍允许开始，并提示本次只根据岗位与面试设置提问。
- 知识来源的“被选中”不等于后端已授权；真实接入时由可信后端执行最终可见性判断。
- 页面底部提供“开始面试”主操作和返回入口。
- 提交期间禁用重复提交并显示进行状态。
- 创建失败时保留全部输入，在原位置展示错误与重试。
- 麦克风准备失败时停留在配置页，不进入正式面试。Mock 阶段仅模拟检查结果，不采集音频。

## 7. 正式面试页

### 7.1 页面布局

页面保留工作台外壳，不切换全屏、深色专用模式或视频会议舞台。

顶部窄栏展示目标岗位、面试类型、当前阶段和时间。时间格式为“已用时间 / 预计时间”。超过预计时长后，已用时间使用危险色，并同时显示“已超时”文字。角落保留低强调的“退出本次练习”。

中间区域为完整对话记录：

- AI 问题和用户回答按时间顺序展示。
- 当前问题提高层级，历史内容保持可读但降低强调。
- AI 文本以流式形式增量追加。
- 用户位于底部附近时自动跟随新内容；用户主动回看历史时不抢夺滚动位置。

底部固定回答区展示麦克风状态、实时音量反馈和只读实时转写。右下角主操作是“结束录音并提交”。没有识别出有效内容时按钮禁用。

### 7.2 回合状态机

```text
preparing
  → interviewer_streaming
  → listening
  → submitting_answer
  → thinking
  → interviewer_streaming
      ├─ listening
      └─ completion_ready
           → report_processing
           → summary
```

页面只允许状态机定义的操作：

- AI 文本流完成后进入持续监听。
- 监听不会因停顿自动提交。
- 用户点击“结束录音并提交”后锁定本轮转写、暂停监听并提交回答。
- 思考和 AI 回复期间不采集下一轮音频。
- AI 回复完成且决定继续时恢复监听。
- AI 发出面试结束指令时停止监听，主按钮切换为“结束面试并查看分析”。

### 7.3 时间与结束规则

- 预计时长不是立即截断当前回答的硬倒计时。
- 超过预计时长后不再提出新问题。
- 用户提交当前回答后自动进入结束流程。
- 正常情况下由 AI 根据覆盖度、题量和进度决定结束，用户不能主动生成一份不完整总结。
- “退出本次练习”是安全出口，需要二次确认；退出不生成报告，也不进入历史记录。

### 7.4 错误恢复

- 麦克风权限丢失、设备不可用或识别断开：暂停计时和交互，保留已完成对话，提供重新连接。
- AI 回复失败：保留已提交回答，只重试当前 AI 回合，不要求用户重新作答。
- 页面卸载、路由切换或确认退出：停止媒体模拟、取消流读取并释放所有订阅和计时器。
- 重连与回放必须等待正式协议；Mock 只模拟有限且确定的恢复行为。

## 8. 总结页

### 8.1 生成状态

点击“结束面试并查看分析”后进入总结路由。报告未完成时，同一路由展示当前生成阶段，不显示虚假的精确百分比。失败时提供“重新生成总结”，成功后替换为报告内容。只有成功报告才使该会话出现在历史列表。

### 8.2 报告结构

1. 结果概览：总评分、目标岗位、类型、难度、完成时间、实际时长和一句总体评价。
2. 能力维度：使用带直接数值标签的横向条形图展示内容相关性、回答结构、专业深度、案例与证据、表达与节奏。
3. 优势与改进重点：各展示 2–4 条来自本次对话的结论。
4. 评分证据：按维度展示问题、回答片段和评分解释，可展开完整上下文。
5. 下一次练习：固定给出 3 条具体、可执行的行动建议。
6. 本次资料：展示被提供给面试 Agent 的知识来源；只有后端返回明确引用证据时才声明某份资料被实际引用。

第一版不使用雷达图或跨场次趋势。报告不得推断人格、情绪或受保护属性。

### 8.3 后续操作

- “返回面试记录”返回 `/interviews`。
- “再练一次”把本次岗位、类型、难度、时长和知识选择带回配置页供用户确认，不直接启动监听。

## 9. 视觉与交互方向

- 复用现有工作台设计令牌、深浅主题、Lucide 图标和公共 UI 组件。
- 视觉参考是 Linear 的克制状态表达、Notion 的安静表单与列表、ChatGPT 的对话可读性；参考其信息组织，不复制品牌外观。
- 每个页面只保留一个主操作；危险或退出操作在视觉和空间上与主操作分离。
- 正式面试页使用收紧的内容宽度和固定回答区建立专注感，不依赖装饰、玻璃效果或数字人。
- 动效只用于状态转换和流式反馈，持续时间保持简短，并保留现有减少动效能力。
- 新增交互保留语义化控件、清晰焦点和可访问名称；状态不能只靠颜色表达。

## 10. 领域模型与 Gateway

继续复用现有 `InterviewGateway`、`KnowledgeGateway`、`UiInterviewSession`、`UiLiveInterviewModel`、`UiInterviewReport`、评分证据和知识来源模型。

为界面补充以下稳定 UI/领域投影：

- `UiInterviewHistoryItem`：历史列表所需的已完成会话摘要。
- `UiInterviewSetupModel`：岗位、场景、默认值和可选知识资料。
- `UiCreateInterviewInput`：页面提交的业务意图，不包含 HTTP DTO 或路径。
- `UiInterviewRuntimeState`：正式面试的可判别联合状态。
- `UiInterviewReportState`：报告未开始、生成中、失败和完成状态。

扩展现有 `InterviewGateway`，以业务意图表达：

- 获取面试首页和已完成记录。
- 获取配置数据并创建会话。
- 获取、订阅和清理实时面试状态。
- 提交当前回答并重试失败回合。
- 放弃练习。
- 请求生成总结并读取报告状态。

知识来源继续由 `KnowledgeGateway` 提供。React 页面只消费 Gateway 和 `Ui*` 模型，不引用具体 Mock 类、后端路由、Electron API 或通用传输函数。

真实接入时，REST DTO、实时控制、媒体传输、流事件解析、运行时校验、鉴权和错误映射全部位于 infrastructure adapter。页面调用方式保持不变。

## 11. Mock 设计

Mock adapter 提供确定性、可测试的行为：

- 多条已完成历史记录与空、失败变体。
- 岗位、场景、知识来源与默认全选配置。
- 创建成功、重复提交防护和创建失败。
- AI 流式提问、持续监听、增量转写、提交、思考和流式回复。
- AI 继续提问、主动结束、超时后结束、媒体失败和当前回合重试。
- 报告生成中、失败和完成。
- 卸载时释放所有计时器、订阅和异步任务。

Mock 不采集或发送真实音频，不创建真实网络连接。界面必须明确标注当前是 Mock/界面演示，不能显示“已连接真实服务”“已上传录音”或“正式评分”等文案。

## 12. 契约与待后端确认项

下列内容未冻结，实现时记录到 `domain/pending.ts` 或既有契约缺口文档，不能在页面或 adapter 中猜测：

- 历史列表、创建会话、放弃会话、报告生成的 method、path、DTO、状态码、错误包装和幂等规则。
- 生产身份、workspace membership、知识选择和最终 EffectiveAccess。
- 音频采集、转写片段、临时与最终文本的格式和生命周期。
- 面试实时控制面与媒体面的握手、鉴权、顺序、去重、重连、恢复窗口、取消和结束语义。
- AI 结束指令的权威来源和事件格式。
- 预计时长与服务端权威结束时间。
- 报告 Job 的进度、失败重试、最终资源和过期行为。
- 评分上下限、版本、归一化和历史报告量表查询。

产品要求使用 100 分制展示。Mock 可以提供明确标注的 100 分制演示数据；真实 adapter 只有在后端提供权威量表上下限或明确归一化规则后，才能展示正式 100 分结果。

现有 `GET /api/v1/agent-runs/{run_id}/events` 只属于 Agent Run SSE，不能被假设为面试流或报告 Job 流。

## 13. 测试与验收

自动化测试覆盖：

- 四条路由、主导航目标和返回行为。
- 历史列表加载、空、失败、倒序排列和完成记录过滤。
- 配置默认值、知识库默认全选、取消选择、无资料、校验和重复提交。
- 正式面试状态机的合法转换和非法操作防护。
- 转写只读、空回答禁用提交和提交后锁定。
- 计时、超时文字与危险色、当前回答完成后结束。
- AI 正常结束后的按钮切换和安全退出不生成报告。
- 流式增量不重复追加，失败回合可重试。
- 页面卸载和路由切换清理计时器、订阅和流。
- 总结生成中、失败、完成、证据展开与再次练习。
- 中英文资源、深浅主题下的结构完整性和基础移动布局。

实现完成后执行：

```text
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
pnpm smoke:desktop
```

手工验收确认 Web 和 Electron 共用页面不访问 Node.js，页面无写死后端地址，没有 Mock 数据泄漏到组件，流与录音模拟在离开页面后停止，所有错误都提供恢复路径。

## 14. 实现顺序

1. 扩展领域模型、`InterviewGateway` 和待确认项。
2. 扩展确定性 Mock 数据、状态机和 Gateway 测试。
3. 增加面试首页与配置页路由和页面测试。
4. 用简洁对话界面替换现有数字人面试舞台，并覆盖生命周期测试。
5. 重构总结页和报告生成状态。
6. 补齐中英文资源、响应式样式和导航。
7. 执行前端门禁与 Web/Electron 手工检查。
